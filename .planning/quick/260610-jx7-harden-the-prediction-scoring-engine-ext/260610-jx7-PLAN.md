Harden the prediction scoring engine without touching the now-verified knockout bracket data/feeders.

Three defects, all knockout-only (group-stage scoring is already correct, idempotent, and live — weeks of runway, but land before knockout results start):

2. The knockout winner bonus derives the user's winner from score only, consulting the explicit `predictedWinner` field exclusively on draw predictions.

Purpose: One trustworthy scoring engine, provably correct via tests, safe under partial failure, before knockout results begin.

CONFIRMED SCORING RULES (lock into tests as source of truth):
- Group: exact score = 3; correct result (home/away/draw) = 1; else 0.
- Knockout: same base (3 / 1 / 0) PLUS +1 if user's predicted winner == actual winner. Perfect knockout pick = 4.
- isBonusMatch: NO scoring effect. By design. Do NOT branch on it in scoring.

Prisma shapes (prisma/schema.prisma):
  model Prediction {
    predictedHome   Int
    predictedAway   Int
    predictedWinner String?   // literal 'home' | 'away' | null
    pointsEarned    Int @default(0)
  }
  model Match {
    stage           String    // 'group' | knockout stages (e.g. 'round32', ...)
    homeTeamId      String?
    awayTeamId      String?
    realScoreHome   Int?
    realScoreAway   Int?
    isBonusMatch    Boolean @default(false)
  }

HOW predictedWinner IS SET IN THE UI (verified — drives the exact shape of fix #2):
  components/MatchCard.tsx (lines 144 & 159), on save:
    predictedWinner: isKnockout && isDraw ? winner : null
  - `winner` is the literal string 'home' or 'away' (a slot, not a team id).
  - So today the explicit predictedWinner field is ONLY populated for knockout DRAW
    predictions. For non-draw knockout predictions it is null and the intended winner
    is implied by predictedHome vs predictedAway.
  - app/api/predictions/route.ts upserts predictedWinner straight through (no transform).
  => Current behavior is consistent with the UI, BUT the bonus must HONOR the explicit
     field whenever it is set ('home'/'away'), and ONLY fall back to score-derived winner
     when predictedWinner is null/empty. This preserves today's draw/penalty handling and
     is robust if the UI later sets predictedWinner on non-draws.

Existing endpoint actual-winner derivation (KEEP this logic in the routes, it sets the
match cascade winner — it is NOT the user's predicted winner):
  - knockout & homeScore > awayScore -> actualWinnerId = match.homeTeamId
  - knockout & awayScore > homeScore -> actualWinnerId = match.awayTeamId
  - knockout draw -> actualWinnerId = body.winnerId (penalties); required (400 if missing in single route)
  - group -> actualWinnerId = null

Test conventions (tests/features/*.test.ts, vitest.config.ts):
  - `import { describe, it, expect } from 'vitest';` — globals enabled, environment 'node'.
  - Path alias '@' -> project root. So `import { calculatePredictionPoints } from '@/lib/scoring';`
  - Tests are PURE LOGIC — no Prisma, no DB, no mocks needed. They import the function and assert.
  - Run: `npm test` (vitest run). Filter: `npx vitest run tests/features/scoring.test.ts`.
